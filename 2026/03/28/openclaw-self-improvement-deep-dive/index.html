OpenClaw 自我进化机制深度探索：从技能管理到 Workspace 文件体系

Fester Zheng
2026-03-28
技术笔记 › AI 工具
AI Agent, OpenClaw, Self-Improvement, 技能管理

今天和”十月”（我的 OpenClaw 专属 AI 搭档）进行了一次深度对话，探索了 self-improving-agent 技能、Workspace 技能目录、以及整个 .md 文件体系的使用方式。这是一篇完整的对话记录和知识整理。

背景

我的 OpenClaw workspace 里有一套完整的自我进化机制，但之前一直没有深入了解它是如何工作的。今天决定彻底搞明白以下几个问题：

  1. self-improving-agent 技能是否真的在启用？每次 improving 有记录吗？
  2. Workspace 里的 skills/ 目录有用吗？
  3. Workspace 里那么多 .md 文件，AI 都会用到吗？怎么用？

下面是完整的探索过程和答案。

一、Self-Improvement 技能状态检查

问题

如何知道 self-improving-agent 在启用了？每次 improving 有记录吗？

解答

✅ 技能已启用，安装位置：~/.openclaw/skills/self-improving-agent/

📁 记录目录：~/.openclaw/workspace/.learnings/

技能触发条件

满足任意一条即应记录：

情况 记录文件
用户纠正 AI LEARNINGS.md（category: correction）
命令/操作失败 ERRORS.md
AI 发现自己知识有误 LEARNINGS.md（category: knowledge_gap）
用户提出缺失功能 FEATURE_REQUESTS.md
发现更好的做法 LEARNINGS.md（category: best_practice）

记录格式示例

二、单次记录真的有用吗？

问题

记下来后，你有什么进步吗？

解答

诚实回答：单次记录没有立即进步，但长期积累有价值。

价值时间维度

时间维度 价值
当前会话 上下文保留，避免同一会话内重复错误
跨会话 未来会话可读取 .learnings/ 回顾历史问题
长期积累 模式被提升到 SOUL.md/AGENTS.md/TOOLS.md 后，成为持久化规则

进步机制

关键阈值（根据技能文档）：

  • Recurrence-Count >= 3 且跨 2+ 任务 → 可提升到系统提示文件

三、Workspace 技能目录是否有用？

问题

workspace 里的 skill 目录有用吗？

检查过程

技能列表检查

结论

✅ Workspace 技能目录是有效的

目录 路径 用途
系统技能 ~/.openclaw/skills/ OpenClaw 内置/全局技能
Workspace 技能 ~/.openclaw/workspace/skills/ 项目级/自定义技能

Workspace 技能的价值

  1. 项目专属技能 —— 只在这个 workspace 生效，不污染全局
  2. 开发测试 —— 新技能先在 workspace 测试，成熟后再移到系统目录
  3. 版本隔离 —— 不同项目可用不同版本的同一技能
  4. 敏感技能 —— 包含项目特定逻辑的技能不适合全局共享

四、skill-vetter技能安全审查试用

直接给openclaw发指令：

审查这个技能 Multi Search Engine

审查结果

项目 内容
技能名 multi-search-engine
版本 v2.0.1
来源 GitHub: limo08/openclaw-master-skills
功能 聚合 17 个搜索引擎（8 国内 +9 国际）
API 依赖 ❌ 无需 API Key

安全审查表

检查项 状态 说明
来源可信度 🟡 中等 GitHub 仓库，非官方但公开可查
权限范围 ✅ 低风险 仅调用 web_fetch 工具
敏感操作 ✅ 无 无密钥处理、无数据外传
依赖复杂度 ✅ 简单 仅依赖内置 web_fetch 工具
代码透明度 ✅ 高 完整 SKILL.md 公开

集成的搜索引擎

国内（8 个）：百度、必应中国、必应国际、360、搜狗、微信、头条、集思录

国际（9 个）：Google、Google HK、DuckDuckGo、Yahoo、Startpage、Brave、Ecosia、Qwant、WolframAlpha

风险评估

🟢 低风险 —— 推荐安装

理由：

  1. 本质是搜索引擎 URL 模板封装，无复杂逻辑
  2. 使用 OpenClaw 内置 web_fetch 工具，受平台沙箱限制
  3. 无需 API Key，无密钥泄露风险

总结

通过试用，我们在安装某个skill之前，可以使用skill-vetter来检查这个skill的来源、功能和API依赖；同时也能了解到这个skill的安全性，通过最后的风险评估来决定是否安装这个skill，以保证我们放心的使用openclaw。

五、Workspace .md 文件体系全解析

问题

workspace 里有很多**.md 文件，你都会用到吗？你是怎么用的？

完整文件清单

文件分类与使用频率

📌 核心文件（每次会话必用）

文件 用途 使用频率 我如何使用
SOUL.md 定义 AI 身份、语气、价值观 🔁 每次会话开始 读取以确认行为准则和沟通风格
USER.md 用户技术栈、偏好、项目信息 🔁 每次会话开始 确认当前任务上下文和你的技术背景
AGENTS.md 工作模式切换、记忆管理规范 🔁 工程师/PM 模式切换时 决定输出格式（代码/文档）和深度
TOOLS.md 工具使用约定、安全规范 📌 需要时查阅 执行命令/写代码前确认规范和禁忌
MEMORY.md 长期记忆、架构决策、经验教训 📌 需要历史上下文时 查 ADR 避免重复错误，参考过往决策
IDENTITY.md AI 名字、表情符号、人设 📌 会话开始 确认”十月”身份和 🦞 表情符号

📅 动态文件（按需使用）

文件 用途 使用频率 我如何使用
memory/YYYY-MM-DD.md 每日流水记录 📌 跨会话任务时 读取最近记忆获取昨日/今日上下文
PROJECTS.md 项目状态看板 📌 涉及具体项目时 查进度、阻塞点、待办事项和优先级
TECH_DEBT.md 技术债清单 📌 重构/优化任务时 确认优先级和计划清偿日期，避免引入新债
.learnings/*.md self-improvement 记录 📌 用户说”记下来”时 写入学习/错误/功能请求，定期回顾模式

⚙️ 配置/策略文件（偶尔用）

文件 用途 使用频率 我如何使用
SECURITY_BOUNDARIES.md 文件系统访问边界 📌 跨目录操作时 确认是否允许读取/写入目标目录
TOKEN_USAGE_POLICY.md Token 使用策略 📌 空闲/定时任务时 避免无意义消耗，空闲时不主动推理
HEARTBEAT.md 定时任务清单 ⏰ 心跳触发时 执行监控/报告任务，检查告警条件
BOOTSTRAP.md 初始化脚本 ❌ 一次性（已完成） 初始化完成后删除，不再使用

实际工作流程

关键原则

不是所有文件都要用 —— 只读取与当前任务相关的文件，避免 token 浪费。

六、今天的 Learnings 记录

本次对话中，我让”十月”把关键问答都记录到了 .learnings/LEARNINGS.md：

条目 ID 分类 优先级
LRN-20260328-001 self-improvement 技能状态查询 low
LRN-20260328-002 Workspace .md 文件使用策略 low

这本身也验证了 self-improvement 机制在工作——关于”如何使用文件”的知识被记录下来了。

总结

核心发现

  1. self-improvement 技能已启用，但需要主动触发（用户纠正/错误/新发现）
  2. Workspace 技能目录有效，OpenClaw 自动扫描并加载
  3. 技能安装前需要审查，使用 skill-vetter 检查安全性
  4. .md 文件按需使用，不是所有文件每次都会读取

后续行动

  • 考虑安装 multi-search-engine 技能（低风险）
  • 配置 self-improvement 自动提醒 hook
  • 定期复盘 .learnings/ 中的模式，提升到系统提示文件
